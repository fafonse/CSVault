When you build software, you need to first ask, what does it need to do?

Requirements can help you solidify that identity during planning.
## Types of requirements
You want to make your high level requirements functional requirements when you can. It makes your project more tangible and actionable.

1. High Level Requirements
	- Vague requirements for what the software does
	- What it is, what it should do
	- You **can't build a program from this**
	- EX: *Students can login to system*
2. Functional requirements
	- Requirements for the software to actually work as intended
	- EX: *Students will be be able to input their D number and password to login through a webpage. Their session will end after they close the tab.*
- Non-functional Requirements
	- Requirement for the program to work for the user
	- Not really *core* to the system
	- EX: *Web page will be optimized for desktop, and must work on all major browsers*
